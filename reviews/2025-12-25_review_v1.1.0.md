**二、問題點 1：`單一 # H1 和 ## H2 之間` 的語法存在歧義**

原文關鍵句：

* 「依據 markdown 格式中單一 `# H1 和 ## H2` 之間，或是兩個 `## H2` 之間，當作一個文本範圍」

這一句在**人類閱讀時可以猜到你的意思**，但對 LLM 來說，有兩個可能的歧義來源：

1. **「# H1 和 ## H2 之間」是指哪一種關係？**

   * 是「某個 H1 與其下一個 H2 之間」？
   * 還是「H1 標題行與 H2 標題行之間的正文」？
   * 是否包含 H1 之下、第一個 H2 之前的所有文字？

2. **「單一」修飾的是什麼？**

   * 單一個 H1？
   * 單一個 H2？
   * 單一個區段？

在語言學上，這是一個**修飾語指向不唯一**的句子。

嚴重程度評估：
**中等**。
多數模型會「猜對」，但在長文件或結構不乾淨的 transcript 上，可能出現邊界處理不一致。

---

**三、問題點 2：`文本範圍` 與後文 `主題範圍` 混用**

你在前半段非常努力定義了「文本範圍」，但在結尾流程控制時，又出現了另一個詞：

* 前段：
  「當作一個文本範圍，一次一個文本範圍做理解」
* 後段：
  「再繼續下一個主題範圍」

對人類來說，這兩個詞你是當同義詞在用；
但對 LLM 來說，**它們是兩個不同的名詞**。

合理但危險的模型推論可能是：

* 文本範圍：結構切片單位（你剛定義的）
* 主題範圍：語義主題（可能跨多個文本範圍）

在某些模型上，這會重新打開「一次處理多個單位」的空間，尤其是在回合切換時。

嚴重程度評估：
**中等偏高**，因為這正好是你先前實測踩到的問題類型。

**四、問題點 3：`保留原本 markdown h1 and h2 title` 的範圍未被限定**

你現在寫的是：

* 「保留原本 markdown h1 and h2 title」

但在 v1.1.0 的語境下，這句話仍然**缺少一個關鍵限定**：

* 是「保留**本次文本範圍所對應的** h1 / h2」？
* 還是「凡是相關的 h1 / h2 都應該保留」？

在你腦中，答案很明確；
但在模型腦中，**它會回到「結構完整性偏好」**，特別是在看到多個 h2 緊鄰時。

這正是你在 v1.0.0 實測中觀察到的行為根源之一，而 v1.1.0 並沒有完全把這條路堵死，只是縮窄了。

嚴重程度評估：
**中等**。不一定每次發生，但發生時會讓你很困惑。

**五、沒有問題、而且值得肯定的地方（這很重要）**

* 「一次一個文本範圍做理解」

也就是說：
